<task type="auto">
  <name>T5.1.2: Implement AlertRouter dedup and rate limit logic</name>
  <files>crates/helm-monitor/src/alerting/mod.rs</files>
  <action>
Implement in `alerting/mod.rs`:

`impl AlertRouter`:
- `fn new(config: AlertConfig) -> Self` — init with empty dedup_state HashMap, Instant::now() for last_sent_time, 0 sent_count
- `fn with_sink(mut self, sink: Box<dyn AlertSink>) -> Self` — builder method to add sink
- `async fn route(&mut self, finding: &Finding) -> Result<(), Box<dyn std::error::Error>>` — main entry point:
  1. Convert Finding → AlertPayload
  2. Check severity gate: if payload.severity < config.min_severity, return Ok(()) (skip silently)
  3. Check dedup: lookup fingerprint in dedup_state; if exists and (now - stored_time) < dedup_window_secs, return Ok(()) (skip)
  4. Check rate limit: if (now - last_sent_time).as_secs_f64() < 60.0 && sent_count >= config.rate_limit_per_min, return Ok(()) (skip)
  5. If all gates pass: fan-out to all sinks concurrently (use tokio::join! or futures::future::join_all)
  6. Update dedup_state[fingerprint] = now and sent_count += 1
  7. Reset sent_count every 60 seconds (if now - last_sent_time > 60, sent_count = 1, last_sent_time = now)

Each sink.send() error is logged but does NOT stop routing to other sinks (log at WARN level).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AlertPayload → AlertSink | Untrusted alert data flows to external sinks; sinks responsible for auth/validation |
| Config file read | Config file may be world-readable; min_severity and rate_limit are not secrets but should not be writable by unprivileged users |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | config file (thresholds.toml) | accept | User controls alerting thresholds; no elevated privilege. File permissions are user responsibility. |
| T-05-02 | Information Disclosure | AlertPayload contents | accept | AlertPayload is derived from Finding; no new PII introduced. Sinks handle confidentiality. |
| T-05-03 | Denial of Service | rate_limit gate | mitigate | Rate limit prevents alert flooding; default 60/min. User can adjust in config. |
| T-05-04 | Spoofing | sink authentication | mitigate | Each sink (webhook, Slack, PagerDuty) validates own auth; documented in sink plans. |
</threat_model>
